# Dynamic Cart Valuation Engine (Elixir + Rust)

Высоконагруженный движок динамического расчета стоимости корзины в реальном времени, спроектированный с использованием гибридной архитектуры **Elixir (OTP/Distributed Orchestration)** и **Rust (NIF/High-Performance Computation)**.

Проект разработан как технологическое решение архитектурной проблемы деградации производительности при расчете сложных, пересекающихся правил лояльности и скидок в условиях экстремальной конкурентной нагрузки (ритейл реального времени / стек ecom.tech).

---

## 1. Актуальность и бизнес-контекст

В e-commerce платформах (таких как Самокат, Мегамаркет или Купер) расчет стоимости корзины — одна из самых критичных и вычислительно сложных операций. При изменении состава корзины система должна в пределах единиц миллисекунд:

1. Проверить десятки пересекающихся промокодов и правил лояльности.
2. Исключить несовместимые акции и рассчитать оптимальную скидку для сотен позиций.
3. Проверить налоговые и региональные ограничения.

### Системная проблема:

- **I/O-bound vs CPU-bound:** Виртуальная машина BEAM идеально оптимизирована для управления миллионами веб-сокетов и параллельных сессий пользователей (I/O-bound). Однако комбинаторные расчеты правил лояльности на чистом Elixir создают высокую нагрузку на CPU (CPU-bound).
- **Деградация задержек (Latency Spikes):** Выполнение тяжелой математики на Elixir-процессах приводит к перегрузке планировщиков (schedulers), росту задержек на хвосте распределения времени отклика (tail latency) и увеличению частоты сборки мусора (GC) из-за копирования термов.

## 2. Архитектура и три сценария оптимизации

Проект реализует паттерн **"Endurance Stack"** и решает три фундаментальных архитектурных вызова:

### Сценарий I: Оптимизация маршалинга памяти (Zero-Copy & Lazy List Iteration)

Для базового расчета стоимости корзины стандартное декодирование карт Elixir (`%{item_id: ..., price: ...}`) требует выполнения дорогостоящих хеш-поисков Erlang VM для каждого ключа.

- **Решение:**
  - Обмен данными переведен на плоские кортежи **`NifTuple`**, транслируемые напрямую в непрерывные массивы памяти C на нативном уровне.
  - Обход списка корзин на стороне Rust выполняется лениво через **`ListIterator`**, что предотвращает аллокацию промежуточных нативных векторов `Vec`.
  - Поля строк-ключей `item_id` декодируются по принципу Zero-Copy с типом **`&'a str`**, ссылаясь напрямую на бинарные данные в куче BEAM без повторного выделения памяти под новые owned-строки `String` [1].

### Сценарий II: Оптимальное комбинаторное дерево поиска (NP-сложная задача / Branch and Bound)

Поиск оптимального сочетания скидок при наличии **конфликтующих (взаимоисключающих) промокодов** представляет собой NP-сложную задачу (аналог задачи о рюкзаке) [7].

- **Решение:**
  - На стороне Rust реализован рекурсивный обход дерева состояний на базе алгоритма **Backtracking (поиск в глубину)**.
  - Проверка конфликтов правил заменена с линейного сканирования векторов на **побитовые маски (`u64`)**. Проверка конфликта нового правила со всеми ранее выбранными выполняется процессором за **1 такт** через инструкцию побитового `AND` (`&`) [5].
  - Внедрен алгоритм **ветвей и границ (Branch and Bound)**: на каждом шаге рекурсии рассчитывается сумма потенциально возможных оставшихся скидок; если текущая сумма дисконта плюс остаток не способны побить уже найденный рекорд, ветвь рекурсии мгновенно отсекается.
  - Все операции выполняются исключительно на стеке без выделения динамической памяти на куче Rust (Zero Allocations) [1].

### Сценарий III: Обход бутылочного горлышка процессов (Lock-Free Shared Registry)

При удержании состояния корзины внутри стандартного `GenServer` параллельные запросы на чтение и расчет стоимости выстраиваются в очередь сообщений (mailbox) процесса, обрабатываясь строго последовательно.

- **Решение:**
  - Применен паттерн **CQRS**: команды на мутацию корзины (запись) идут через `GenServer` (`CartServer`), а конкурентные запросы на чтение и расчет стоимости идут напрямую в обход очереди сообщений.
  - На стороне Rust создан разделяемый ресурс `SharedCartRegistry` на базе потокобезопасной сегментированной хеш-карты **`DashMap`** [7]. Elixir-процессы читают состояние корзин напрямую из DashMap в обход очередей сообщений Erlang-процессов.

## 3. Распределенный кластер и локальность данных (Data Locality)

Для работы в масштабах распределенной сети (Kubernetes) проект расширен до отказоустойчивого кластера с поддержкой автоматического объединения нод и динамической маршрутизации запросов [13].

```text
               [Запрос на расчет корзины "cart_1" пришел на node3]
                                      │
                                      ▼
                        ┌───────────────────────────┐
                        │      Horde.Registry       │ (Где живет "cart_1"?)
                        └─────────────┬─────────────┘
                                      │
                                      ▼
                     (Корзина "cart_1" живет на node1)
                                      │
                                      ▼
                        ┌───────────────────────────┐
                        │         :rpc.call         │ (Вызов расчета на node1)
                        └─────────────┬─────────────┘
                                      │
                                      ▼
                       ┌─────────────────────────────┐
                       │  Rustler NIF на node1       │ (Чтение из локального DashMap)
                       └──────────────┬──────────────┘
                                      │
                                      ▼
                             (Результат f64)
                                      │
                                      ▼
                         [Возврат ответа на node3]
```

### Ключевые компоненты кластера:

1. **`libcluster` (Node Discovery):** Обеспечивает автоматическое сетевое обнаружение контейнеров в приватной мостовой сети Docker и их объединение в единую распределенную Erlang-сеть [11, 13, 17].
2. **`Horde` (Distributed Registry & Supervisor):** Распределенный реестр и динамический супервизор процессов на базе CRDT [16]. `Horde` гарантирует уникальность процесса `CartServer` в рамках всего кластера и равномерно распределяет процессы по выжившим нодам [16].
3. **Распределенный CQRS-маршрутизатор (`DistributedRouter`):**
   - **Запись:** Прозрачно направляет изменения на ноду-владелец конкретного `CartServer` [16].
   - **Чтение (RPC-оптимизация):** Если процесс корзины находится на удаленной ноде, роутер делает сетевой RPC-вызов `:rpc.call` на целевую ноду [11, 14, 15]. При этом сырой список правил `raw_rules` безопасно сериализуется, передается по сети и **компилируется на месте на целевой ноде** [11, 15]. Это исключает пересылку тяжелых C-указателей NIF-ресурсов через сеть (что привело бы к ошибке `badarg`) и предотвращает сетевой перенос сырых данных корзин [15].

## 4. Системный анализ и качество кода (QA Pipeline)

Проект интегрирует строгий промышленный стандарт статического анализа и безопасности, принятый в ecom.tech.

### Конвейер статических проверок Elixir:

- **Credo (Строгий режим):** Проверка архитектурных антипаттернов, стиля и структуры модулей.
- **Dialyxir (Dialyzer):** Строгий статический анализ типов на основе спецификаций `@spec`.

### Гарантии безопасности Rust (Clippy & Compiler):

Паника (`panic!`) в неуправляемом коде NIF приводит к мгновенному аварийному завершению всей Erlang VM. Для обеспечения стабильности на уровне компиляции настроены жесткие ограничения:

- **Запрет паникующих макросов:** `unwrap_used = "deny"`, `expect_used = "deny"`, `panic = "deny"`. Любая обработка ошибок возвращает `Result<T, E>` в Elixir.
- **Запрет явного разыменования NIF-ресурсов:** `clippy::explicit-auto-deref` — гарантирует использование встроенного механизма безопасного автоматического разыменования умных указателей `ResourceArc`.
- **Защита от сбоя тредов:** `indexing_slicing = "deny"` — предотвращает выход за границы векторов и слайсов (out-of-bounds).
- **Прохождение panic-границы:** Для структуры `SharedCartRegistry` вручную имплементированы маркерные типажи `RefUnwindSafe` и `UnwindSafe` для безопасной передачи ресурса через границу `catch_unwind` в Rustler [9].

## 6. Установка и запуск

### Требования к окружению:

- **macOS** (Apple Silicon) или **Linux**
- **Erlang/OTP 28+** & **Elixir 1.19.5+**
- **Rust 1.80+** (компилятор `rustc`, менеджер `cargo`)
- **OrbStack** или **Docker** (для запуска кластера) [5, 13]

### 1. Локальная сборка проекта и тестирование QA-пайплайна

```bash
# Установка Elixir-зависимостей
mix deps.get

# Проверка качества кода (CI Simulation)
make ci

# Запуск тестов (включая сетевые интеграционные тесты)
mix test
```

### 2. Запуск распределенного кластера из 3 нод в Docker (через OrbStack)

```bash
# Сборка и старт кластера
docker compose up --build
```

_Контейнеры `cart_node_1`, `cart_node_2` и `cart_node_3` запустятся, сформируют Erlang-сеть и автоматически синхронизируют реестры Horde [13, 16, 17, 18]._

### 3. Интерактивное подключение к ноде кластера (Remote Console)

Вы можете войти в интерактивную консоль работающего контейнера прямо с хост-машины [11, 13]:

```bash
docker exec -it cart_node_1 bin/cart_engine remote
```

#### Разбор результатов бенчмарка III:

Устранение флага `schedule = "DirtyCpu"` (который добавлял `~15 мкс` на переключение потоков ОС для сверхбыстрых задач) и распределение запросов по сегментам `DashMap` (исключившее Cache Line Bouncing процессора) позволило нативному конкурентному чтению показать прирост производительности **в 4.34 раза (на 334%) по сравнению с GenServer** [7, 10]. Медианное время расчета составило рекордные **`2.50 μs`** [7].

#### Системный вывод по предотвращению GC-давления (Latency p99):

При пиковой нагрузке в **100 000 RPS** (запросов в секунду):

- **В Elixir:** Система генерирует **~588 МБ временной памяти в секунду** на куче Erlang-процессов. Это приводит к непрерывному запуску сборщика мусора (GC) во всех потоках шедулеров, вызывая просадки по латентности ( latency spikes) на 99.9-м процентиле.
- **В Rust (NIF):** Вычисления генерируют **0 байт мусора** на Erlang-куче [1]. График времени отклика остается стабильным и плоским, гарантируя предсказуемый tail latency в рамках SLA.
